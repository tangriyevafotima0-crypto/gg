# Image extraction and embedding for the Uzbek chess book PDF

Two scripts extract chess diagram images from the English source PDF and embed them into the generated Uzbek translation PDF. `extract_images.py` pulls 256 diagrams, 26 chapter dividers, and 13 icons using PyMuPDF's xref-based image enumeration, filtering by pixel dimensions. `generate_pdf_with_images.py` rebuilds the Uzbek PDF using fpdf2, replacing `[Diagramma X.Y]` text placeholders with the corresponding extracted PNG files, and inserting full-page dividers before each numbered chapter.

**Watch for:** 33 chess diagrams are silently dropped in the output PDF (confirmed). Lines containing two placeholders like `[Diagramma 12.4] [Diagramma 12.5]` only have the first one rendered because the regex uses `match()` and then `continue`s to the next line. The verification passed only because nobody noticed the count was 223 diagrams, not 256.

---

## High-level view

The extraction side is correct — per-chapter diagram counts in the extracted metadata match the total placeholders in the Uzbek text (256) when both items on multi-placeholder lines are counted. The dimension heuristic cleanly separates chess diagrams from the other image types in the PDF.

The PDF generation has a single blocking defect: `_write_content` processes lines one at a time and the diagram-matching regex uses `re.match()` which anchors at line start. When the Uzbek source text places two diagram references on one line (33 occurrences across 8 chapters), only the first is rendered. The second diagram image exists on disk but never appears in the output. The fix is to use `findall` or `finditer` on each line and insert all matched diagrams.

The page-break logic for images is manual (comparing remaining page height against calculated image height). Once the multi-placeholder bug is fixed, two wide diagrams from the same line could produce a page break after the first that doesn't account for the second — the height check would need to run per-diagram rather than once per line.

## Multi-placeholder line handling drops 33 diagrams

In `generate_pdf_with_images.py`, `_write_content` processes lines sequentially:

```python
match = DIAGRAM_PATTERN.match(line)
if match:
    ch_num = match.group(1)
    diag_num = match.group(2)
    self._insert_diagram(ch_num, diag_num)
    i += 1
    continue
```

`re.match()` only tests the start of the string. For a line like `[Diagramma 12.4] [Diagramma 12.5]`, it matches `12.4`, inserts that image, then `continue`s — discarding `12.5` entirely.

Affected chapters and their missed diagram counts:

| Chapter | Extracted | Actually rendered | Missed |
|---------|-----------|-------------------|--------|
| 12 | 8 | 7 | 1 |
| 14 | 28 | 23 | 5 |
| 18 | 22 | 17 | 5 |
| 19 | 17 | 16 | 1 |
| 21 | 9 | 7 | 2 |
| 24 | 21 | 16 | 5 |
| 25 | 39 | 29 | 10 |
| 26 | 15 | 11 | 4 |

Total: 33 diagrams never appear in the PDF despite being correctly extracted and present on disk.

The fix: replace the `match` + `continue` block with `finditer`:

```python
matches = list(DIAGRAM_PATTERN.finditer(line))
if matches:
    for m in matches:
        self._insert_diagram(m.group(1), m.group(2))
    i += 1
    continue
```

## Chapter 11 title mismatch

```python
("11_check.txt", "11-bob. Shoh", 11),
```

"Shoh" means "King" — identical to chapter 10's title. The file `11_check.txt` covers the "check" concept. The correct title should be "11-bob. Shah" or the Uzbek equivalent for "Check".

## Image height and page-break sequencing

`_insert_diagram` computes `img_h` from the pixel aspect ratio, checks if the image fits, and if not, calls `self.add_page()`. The subsequent `self.ln(4)` runs unconditionally:

```python
if needed_space > available:
    self.add_page()

self.ln(4)  # Always runs, even after a fresh page break
```

Diagrams that triggered a page break get 4mm of dead space below the header at the top of the fresh page, while diagrams that fit on the current page get 4mm relative to the preceding content. The inconsistency is cosmetic but visible on chapters with many diagrams.

## Shared xref fragility in extraction

`processed_xrefs` prevents double-counting if the same image xref appears across chapter page ranges. Since diagrams are numbered sequentially per chapter, any xref collision would silently reduce a later chapter's count and produce a numbering mismatch with the Uzbek placeholders. This hasn't triggered (counts are correct), but the assumption that every chess diagram has a unique xref is implicit and unvalidated.
